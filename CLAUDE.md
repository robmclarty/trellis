# Trellis — Plugin Context

Trellis is a Claude Code plugin for **spec-driven development**. It provides 10 composable skills that take a feature from problem framing through implementation.

## Philosophy

Every feature goes through structured phases. Each phase produces a durable markdown artifact in the specs directory (`.specs/` by default, configurable via `trellis.json`). Downstream phases inherit upstream artifacts — the spec references the pitch, the plan references the spec, tasks reference the plan, and implementation follows the tasks.

## Skill relationships

```text
Foundation:     guidelines ─────────────────────────────────────────────┐
                sketch ─────────────────────────┐                       │
                                                │                       │
Pipeline:       pitch → spec → clarify → compliance → plan → tasks      │
                  │       │       │          │          │       │       │
                  └───────┴───────┴──────────┴──────────┴───────┘       │
                          ↑ all inherit from guidelines ────────────────┘

Orchestration:  pipeline (runs pitch → tasks in one session)
Execution:      implement (turns tasks into working code)
```

- **Guidelines** and **Sketch** are foundational — run before starting a feature pipeline
- **Pitch → Spec → Clarify → Compliance → Plan → Tasks** is the specification pipeline, each building on the previous
- **Pipeline** orchestrates the spec pipeline as a single session (interactive or auto mode)
- **Implement** executes tasks through a check-driven feedback loop with an optional test-writer (TDD) and LLM judge

## Key design decisions

- **`context: fork`**: Clarify and Compliance run in isolated contexts. They cannot ask the user questions mid-run. The pipeline skill gathers user input upfront and embeds it in the spec before invoking forked skills.
- **`{specsDir}/{feature}/tasks.json`**: Single file serving as both task definition and execution state. Generated by `/trellis:tasks`, updated in-place during implementation. Re-running `/trellis:tasks` overwrites tasks.json, resetting all status to "pending". Each feature has its own tasks.json, enabling parallel implementations.
- **Single check command**: A CLI chain (e.g., `npm run lint && npm run typecheck && npm run build && npm run test`) that replaces the old oracle pipeline. Configured in guidelines.md, propagated to tasks.json. The `&&` chain provides stop-on-failure for free.
- **Ralph mode (Docker)**: Each task runs in a fresh Claude Code context inside a Docker container with `--dangerously-skip-permissions`. Docker is the security boundary. The loop script (`scripts/ralph-loop.sh`) does ALL orchestration — it assembles prompts from templates (`scripts/templates/`) via `assemble-prompt.py` and sends them to `claude -p` directly. The implement skill is NOT invoked inside Docker. The check command runs on the HOST using the host's toolchain. Supports `--stream`, `--tail`, and `--no-judge` output modes.
- **Test-writer per task**: Before implementation, `should-write-tests.py` (deterministic heuristic) decides if the test-writer agent should run. Behavioral tasks get tests; structural tasks skip. This makes `check` meaningful because it includes tests.
- **Judge at end**: Runs once after all tasks complete (default-on, opt-out with `--no-judge`). Evaluates intent alignment — did you build what the spec asked for, not just what the tasks described.
- **Agents**: Generation agents (pitch-writer, spec-writer, plan-writer, task-writer, guidelines-writer, sketch-writer) run on Sonnet for cost-efficient document generation. Judge and Test Writer run on Opus for analytical tasks. All defined in `agents/` and spawned by their parent skills.
- **Scripts**: Deterministic work (prereq validation, marker extraction, pipeline status, prompt assembly, test-writer heuristic, task status updates) lives in `scripts/` as standalone Python scripts that output JSON. Skills call these in pre-flight; the ralph loop calls them for orchestration.
- **Hooks**: Python-based validators in `hooks/` for spec structure, ambiguity markers, task status, and session startup status. Two-layer architecture: hooks handle orchestration (stdin JSON, relevance filtering, formatting), scripts handle data processing.
- **`trellis.json`**: Stores the configured specs directory (`specsDir`). Created by `/guidelines` on first run. All skills and hooks read from it, falling back to `.specs/` if absent.
