## Instructions

### Step 1: Review the implementation

Use Glob, Grep, and Read to examine the completed implementation. Focus on:

- Files listed in the plan's §6 (File Structure)
- Functions and components referenced in tasks marked "done"
- Hot paths identified by the spec's interface definitions

### Step 2: Identify optimization opportunities

Look for these categories of optimization:

1. **Function-level simplification** — Functions with unnecessary complexity, redundant branches, or verbose logic that can be expressed more concisely.
2. **Redundant computation** — Repeated calculations, unnecessary data transformations, or work done multiple times when once would suffice.
3. **Component efficiency** — Components doing more work than needed per render cycle, unnecessary re-renders, or expensive operations that could be memoized.
4. **Cross-module interactions** — Unnecessary serialization/deserialization, redundant data passing, or chatty interfaces that could be batched.

### Step 3: Create tasks in tasks.json

For each optimization opportunity (maximum 5), create a new task. Find the highest existing phase number N and use phase N+1.

Each new task must have:

- `"id"`: "{N+1}.{task_number}" (e.g., "4.1", "4.2")
- `"phase"`: N+1
- `"title"`: Short, verb-first description (e.g., "Memoize expensive calculation in ProductList")
- `"do"`: Concrete optimization instructions referencing specific files and functions. Include what the current code does and what the optimized version should do.
- `"verify"`: How to confirm the optimization preserves behavior (e.g., "All existing tests pass. No functional changes.")
- `"parallel"`: false
- `"status"`: "pending"
- `"iteration"`: null
- `"source"`: "optimizer"

**Hard constraints:**
- Do NOT propose architectural changes (no new layers, no service extraction, no schema changes)
- Do NOT add new features — optimizations must preserve existing behavior exactly
- Maximum 5 tasks. Prioritize by impact.
- If no meaningful optimizations exist, do NOT create tasks. Write the empty section to plan.md instead.

Write the updated tasks.json to the same file path. Preserve the exact JSON structure with `feature`, `check`, `judge`, `redefinitionPass`, and `tasks` fields. Do not modify existing tasks.

### Step 4: Update plan.md

Append a new section to the end of plan.md:

```markdown

```

If no optimizations were identified, append:

```markdown

_No optimizations identified. Implementation is clean._
```

Do NOT run check commands, install dependencies, or make git commits. Only read project files for analysis and write the updated tasks.json and plan.md.
