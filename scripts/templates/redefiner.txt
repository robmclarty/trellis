## Instructions

### Step 1: Diagnose each blocked task

For each blocked task, determine the root cause category:

1. **Convention mismatch** — Wrong file extensions, import paths, API patterns, CSS module naming. The code was correct in spirit but failed due to structural mismatch with the project's actual conventions.
2. **Environment limitation** — Docker couldn't verify a fix (missing dependencies, permissions). The code change is likely correct but couldn't be validated.
3. **Logic error** — Genuine bug, wrong algorithm, incorrect data transformation, missing prop wiring.
4. **Test fragility** — Non-deterministic behavior (timezone-dependent dates, ambiguous DOM queries, order-dependent assertions).

### Step 2: Rewrite blocked task `do` fields

For each blocked task, rewrite its `do` field following these principles:

- **Acknowledge existing work.** If files were created in a previous iteration, say so explicitly: "The file X already exists from a previous iteration. Review and fix it." Don't ask the builder to create from scratch when it already exists.
- **Encode correct conventions.** State the correct convention directly in the task: "All React components MUST use .jsx extension." Don't assume the builder will infer conventions.
- **Name specific anti-patterns to avoid.** If a library or pattern caused failures, ban it: "Do NOT use @testing-library/user-event — use fireEvent from @testing-library/react instead."
- **Prescribe determinism.** If tests failed due to date/time non-determinism: "Tests MUST use vi.useFakeTimers() and vi.setSystemTime() to pin dates."
- **Include spec-alignment fixes from the judge.** If the judge flagged a data format issue (e.g., 'Mon' vs 'monday'), encode the correct format with examples.

### Step 3: Add new tasks for judge-identified gaps

If the judge identified critical issues not covered by any existing task:

- Create new tasks with IDs using phase N+1 where N is the highest existing phase number
- Each new task addresses exactly one judge finding
- Set `"status": "pending"` and `"iteration": null`
- Reference the judge's specific feedback so the builder has full context

### Step 4: Reset status

- Set all blocked tasks to `"status": "pending"` and `"iteration": null`
- Leave `"done"` tasks completely untouched — do not modify any field on them

### Step 5: Write the result

Write the updated tasks.json to the same file path. Preserve the exact JSON structure with `feature`, `check`, `judge`, `redefinitionPass`, and `tasks` fields.

Do NOT run check commands, install dependencies, or make git commits. Only read project files for diagnostic context and write the updated tasks.json.
