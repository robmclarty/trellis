## Instructions

### Step 1: Analyze the build logs

Study the build logs for evidence of:

- Tasks that required retries — the pattern is likely brittle
- Check failures and their error messages — edge cases may be unhandled
- Patterns the builder struggled with — may indicate fragile design
- Judge feedback about quality or completeness gaps

### Step 2: Review the implementation

Use Glob, Grep, and Read to examine the areas flagged by the logs. Cross-reference with the spec to understand expected behavior. Focus on:

1. **Fragile patterns** — Code that required retries during the build. If the builder struggled, the pattern is brittle.
2. **Edge case gaps** — Situations revealed by check failures that suggest missing edge case handling.
3. **Error handling completeness** — Thin error paths (generic catch blocks, missing cleanup, swallowed errors).
4. **Robustness patterns** — Missing input validation, boundary conditions, or defensive checks.

### Step 3: Create tasks in tasks.json

For each improvement opportunity (maximum 5), create a new task. Find the highest existing phase number N (which may include optimizer-added phases) and use phase N+1.

Each new task must have:

- `"id"`: "{N+1}.{task_number}" (e.g., "5.1", "5.2")
- `"phase"`: N+1
- `"title"`: Short, verb-first description (e.g., "Add boundary validation to date parser")
- `"do"`: Concrete improvement instructions referencing specific files, functions, AND the log evidence that motivated the change. Example: "The retry log for task 2.3 shows that empty input caused an unhandled TypeError. Add input validation to parseDate() in src/utils/date.ts."
- `"verify"`: How to confirm the improvement works (e.g., "Existing tests pass. New edge case is handled without throwing.")
- `"parallel"`: false
- `"status"`: "pending"
- `"iteration"`: null
- `"source"`: "improver"

**Hard constraints:**
- Stay within the spec's scope. Do NOT add new features, endpoints, UI elements, or capabilities.
- Do NOT duplicate optimizer tasks. Check the current tasks.json for optimizer-added tasks and address different concerns.
- Maximum 5 tasks. Prioritize by impact.
- If no meaningful improvements exist, do NOT create tasks. Write the empty section to plan.md instead.

Write the updated tasks.json to the same file path. Preserve the exact JSON structure with `feature`, `check`, `judge`, `redefinitionPass`, and `tasks` fields. Do not modify existing tasks.

### Step 4: Update plan.md

Append a new section to the end of plan.md:

```markdown
## §12 — Improvement Additions (auto-generated)

_Added by improver agent after build completion. Based on build log analysis._

- **{id}: {title}** — {one-line rationale, citing log evidence}
```

If no improvements were identified, append:

```markdown
## §12 — Improvement Additions (auto-generated)

_No improvements identified. Implementation is robust._
```

Do NOT run check commands, install dependencies, or make git commits. Only read project files for analysis and write the updated tasks.json and plan.md.
