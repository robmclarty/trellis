# The Implementation Loop

This document explains how `trellis:build` turns spec artifacts into working code.

## Overview

The build skill reads `tasks.json` (generated by `/trellis:prep`) and executes each task through a check-driven feedback loop. Two execution modes exist:

- **In-session mode** — The SKILL.md orchestrates within the user's conversation. Good for small features.
- **Ralph mode** — A Docker-based loop script runs each task in a fresh Claude Code context. Good for large features where context degradation is a concern.

Both modes follow the same overall pattern:

1. **Task loop** — For each pending task: optionally write tests → implement → check → mark done or blocked.
2. **Judge** — Evaluate intent alignment against the spec.
3. **Redefinition loop** — If tasks are blocked or the judge finds gaps, rewrite task definitions and re-run. Up to 3 passes.
4. **Polish phase** — Optimizer and improver agents review the finished build for performance and robustness improvements, then execute any new tasks in a final pass.

## tasks.json

The single file serving as both task definition and execution state. Generated by the task-writer agent via `/trellis:prep`, updated in-place during implementation.

```json
{
  "feature": "team-kudos",
  "check": "npm run lint && npm run typecheck && npm run build && npm test",
  "judge": true,
  "redefinitionPass": 0,
  "tasks": [
    {
      "id": "1.1",
      "phase": 1,
      "title": "Create data model",
      "do": "Create the SQLAlchemy model...",
      "verify": "Model imports without errors...",
      "status": "pending",
      "iteration": null
    }
  ]
}
```

Key fields:

- **check** — The CLI command chain that must exit 0 for code to be correct. Comes from guidelines.md.
- **status** — `pending`, `done`, or `blocked`. Updated in-place by `update-tasks.py`.
- **judge** — Whether to run the judge agent at the end (default true).
- **redefinitionPass** — Counter tracking how many times the redefiner has rewritten tasks (0 = never redefined).
- **source** — Optional field on tasks added by the optimizer (`"optimizer"`) or improver (`"improver"`) agents, for traceability.

Re-running `/trellis:prep` overwrites tasks.json, resetting all status to "pending". This is the reset mechanism.

## The Task Loop

For each task where `status == "pending"`:

### Step 1: Test Writer (conditional)

`should-write-tests.py` runs a deterministic keyword heuristic on the task's `verify` text. Behavioral tasks ("rejects invalid input", "returns error when...") get tests written before implementation. Structural tasks ("file exists", "scaffold created") skip this step.

The test-writer is a separate LLM invocation — it cannot see the implementation. This prevents confirmation bias: the tests encode expected behavior independently of the code.

The tests **must fail** initially. That's the point — they define the target. When the builder's code makes them pass via the check command, the task is done.

### Step 2: Implementation

The builder receives the full task context:

- What to build (`do` field)
- Acceptance criteria (`verify` field)
- Architectural guidance (plan.md)
- Project conventions (guidelines.md)
- What's already been built (completed task titles)

The builder writes code only. No orchestration, no state management.

### Step 3: Check

The check command runs **on the host** (not in Docker), using the host's toolchain. A typical check command:

```bash
npm run lint && npm run typecheck && npm run build && npm test
```

The `&&` chain provides stop-on-failure for free — if lint fails, tests don't run. This replaces the old 7-stage oracle pipeline with a single deterministic command.

### Step 4: Pass / Retry / Blocked

- **Pass** → `update-tasks.py` marks the task `done`, git commits progress.
- **Fail** → One retry attempt. The retry prompt includes the check output so the LLM knows exactly what broke. The retry prompt is focused: "fix ONLY the errors shown, don't refactor or add features."
- **Still failing after retry** → Mark `blocked`, git commit, move to next task.

## Judge Review

After all tasks complete (or all remaining are blocked), the judge agent runs once. It evaluates **intent alignment**: did you build what the spec asked for, not just what the tasks described?

The judge receives:

- The full spec
- All tasks with their status
- A git diff stat of changes made

It outputs a verdict: `PASS`, `PARTIAL`, or `FAIL`.

The judge is default-on. Opt out with `--no-judge`.

## Redefinition Loop

If the judge verdict is `PARTIAL` or `FAIL`, or tasks are blocked, the redefiner agent diagnoses what went wrong and rewrites task definitions so a fresh builder can succeed.

### How it works

1. The redefiner reads the judge's verdict, build logs, and blocked task context.
2. For each blocked task, it categorizes the root cause:
   - **Convention mismatch** — Task described patterns that conflict with the project's actual conventions.
   - **Environment limitation** — Docker couldn't verify the fix (missing deps, permissions).
   - **Logic error** — Genuine bug or spec mismatch.
   - **Test fragility** — Non-deterministic behavior (timezone math, ambiguous DOM queries).
3. It rewrites each blocked task's `do` field with lessons learned: acknowledging existing files, encoding correct conventions, naming anti-patterns to avoid, prescribing determinism fixes.
4. If the judge identified gaps not covered by any task, the redefiner adds new tasks.
5. All blocked tasks are reset to `pending`. Done tasks are never touched.
6. The outer loop re-executes: task loop → judge → redefinition decision.

### Limits

Maximum 3 redefinition passes. After that, remaining issues are escalated to the user. The `redefinitionPass` field in tasks.json tracks the current count.

### The outer loop

The task loop, judge, and redefiner form a feedback loop:

```
        ┌───────────────────────────────────────────┐
        │                                           │
        ▼       no more             contains        │
  ┌──────────┐  pending  ┌───────┐  blocked   ┌────────────┐
  │Task Loop │──────────▶│ Judge │───────────▶│ Redefiner  │
  └──────────┘           └───────┘            └────────────┘
        │                    │                      │
        │                  PASS &              limit reached
        │                no blocked             (3 passes)
        ▼                    │                      │
    next task                ▼                      ▼
                         polish or                escalate
                           finish                 to user
```

Each pass through the outer loop starts a fresh task loop that picks up all pending tasks (including rewritten ones), runs the judge again, and decides whether another redefinition pass is needed.

## Polish Phase

After the build is functionally complete — either the judge passed with no blocked tasks, or the redefinition limit was reached — the polish phase runs. Two review agents execute sequentially:

### Optimizer

Reviews the finished implementation for localized performance and simplification opportunities:

- Function-level simplification (unnecessary complexity, redundant branches)
- Redundant computation (repeated calculations, unnecessary transforms)
- Component efficiency (unnecessary re-renders, expensive unmemoized operations)
- Cross-module interactions (chatty interfaces, redundant serialization)

Hard constraints: no architectural changes, no new features, max 5 tasks.

The optimizer appends §11 to plan.md documenting its additions.

### Improver

Reviews the implementation AND build logs to identify robustness improvements based on what was learned during the build:

- Fragile patterns (code that required retries during the build)
- Edge case gaps (situations the builder encountered that revealed missing handling)
- Error handling completeness (thin catch blocks, swallowed errors)
- Robustness patterns (missing input validation, boundary conditions)

Hard constraints: stays within spec scope, no feature additions, max 5 tasks, no overlap with optimizer tasks.

The improver appends §12 to plan.md documenting its additions.

### Execution

If either agent created new tasks, they're executed in one final task pass — same task loop as above, but without a subsequent judge or redefinition cycle. The polish phase runs at most once.

Opt out with `--no-polish`. Skipped automatically if the judge was disabled (`--no-judge`).

## In-Session Mode

`/trellis:build <feature>`

The SKILL.md orchestrates within the user's conversation:

1. Pre-flight: read tasks.json, validate prerequisites
2. For each pending task: spawn test-writer agent → implement → run check → update tasks.json
3. After all tasks: spawn judge agent

This mode is interactive — the user can see progress and intervene. Good for features where you want to stay in the loop.

## Ralph Mode (Docker)

`/trellis:build <feature> with ralph [--stream|--tail|--no-judge|--no-polish]`

The loop script (`scripts/ralph-loop.sh`) does ALL orchestration. The build SKILL.md is NOT invoked inside Docker. The full sequence:

### Per task

1. **`should-write-tests.py`** — Deterministic heuristic: should the test-writer run?
2. **`assemble-prompt.py test-writer`** → assembled prompt sent to `claude -p` in Docker
3. **`assemble-prompt.py builder`** → assembled prompt sent to `claude -p` in Docker
4. **Check command** runs on host
5. **`update-tasks.py`** — marks task done or blocked
6. **Git commit** — each task is committed individually

### After all tasks

7. **`assemble-prompt.py judge`** → judge runs in Docker, outputs verdict
8. **Redefinition decision** — If `PASS` with no blocked tasks → done. Otherwise → redefiner.
9. **`assemble-prompt.py redefiner`** → rewrites blocked tasks, adds new tasks for gaps
10. **Loop back to step 1** — re-execute pending tasks (up to 3 redefinition passes)

### After build completes

11. **`assemble-prompt.py optimizer`** → reviews implementation, may add tasks
12. **`assemble-prompt.py improver`** → reviews build logs, may add tasks (sees optimizer's additions)
13. **Execute polish tasks** — one final task pass if new tasks exist

Each Docker invocation gets a fresh context window. The LLM receives a fully-baked prompt with all context pre-assembled. No skill loading, no orchestration instructions, no state management.

### Output modes

| Flag | Behavior |
|------|----------|
| *(default)* | Silent — output goes to log files only. Status shown between tasks. |
| `--stream` | Full Claude output visible in real-time via `tee`, also logged. |
| `--tail` | Silent during task, shows last 50 lines of log after each task completes. |

### Authentication

Two modes, auto-detected:

- **API key** — Set `ANTHROPIC_API_KEY` env var.
- **OAuth/subscription** — Run `ralph-loop.sh --login` once. Credentials stored in a Docker volume and reused across tasks.

### Resume from interruption

Kill the process, re-run `/trellis:build <feature> with ralph`. The script reads tasks.json and picks up from the first pending task. There is no separate state file — tasks.json on disk IS the resume point.

## Prompt Templates

Seven plain-text files in `scripts/templates/` with `{{variable}}` placeholders:

- **test-writer.txt** — Writes failing tests for a task's verify criteria
- **builder.txt** — Writes code for one task
- **builder-retry.txt** — Fixes code after check failed
- **judge.txt** — End-of-run intent alignment review
- **redefiner.txt** — Diagnoses blocked tasks and rewrites definitions
- **optimizer.txt** — Reviews implementation for performance/simplification
- **improver.txt** — Reviews build logs for robustness improvements

`assemble-prompt.py` resolves the variables from spec artifacts (tasks.json, plan.md, spec.md, guidelines.md) and outputs the fully assembled prompt as JSON.

## When to Use Which Mode

| Scenario | Mode |
|----------|------|
| Small feature (1-3 tasks) | In-session |
| Want to stay in the loop | In-session |
| Large feature (5+ tasks) | Ralph |
| Fire-and-forget overnight | Ralph |
| Context degradation concern | Ralph |
| Docker not available | In-session |
