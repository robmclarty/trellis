# The Implementation Loop

This document explains how `trellis:implement` turns spec artifacts into working code.

## Overview

The implement skill reads `tasks.json` (generated by `/trellis:prep`) and executes each task through a check-driven feedback loop. Two execution modes exist:

- **In-session mode** — The SKILL.md orchestrates within the user's conversation. Good for small features.
- **Ralph mode** — A Docker-based loop script runs each task in a fresh Claude Code context. Good for large features where context degradation is a concern.

Both modes follow the same per-task pattern: optionally write tests → implement → run check → mark done or blocked.

## tasks.json

The single file serving as both task definition and execution state. Generated by the task-writer agent via `/trellis:prep`, updated in-place during implementation.

```json
{
  "feature": "team-kudos",
  "check": "npm run lint && npm run typecheck && npm run build && npm test",
  "judge": true,
  "tasks": [
    {
      "id": "1.1",
      "phase": 1,
      "title": "Create data model",
      "do": "Create the SQLAlchemy model...",
      "verify": "Model imports without errors...",
      "status": "pending",
      "iteration": null
    }
  ]
}
```

Key fields:
- **check** — The CLI command chain that must exit 0 for code to be correct. Comes from guidelines.md.
- **status** — `pending`, `done`, or `blocked`. Updated in-place by `update-tasks.py`.
- **judge** — Whether to run the judge agent at the end (default true).

Re-running `/trellis:prep` overwrites tasks.json, resetting all status to "pending". This is the reset mechanism.

## The Task Loop

For each task where `status == "pending"`:

### Step 1: Test Writer (conditional)

`should-write-tests.py` runs a deterministic keyword heuristic on the task's `verify` text. Behavioral tasks ("rejects invalid input", "returns error when...") get tests written before implementation. Structural tasks ("file exists", "scaffold created") skip this step.

The test-writer is a separate LLM invocation — it cannot see the implementation. This prevents confirmation bias: the tests encode expected behavior independently of the code.

The tests **must fail** initially. That's the point — they define the target. When the implementor's code makes them pass via the check command, the task is done.

### Step 2: Implementation

The implementor receives the full task context:
- What to build (`do` field)
- Acceptance criteria (`verify` field)
- Architectural guidance (plan.md)
- Project conventions (guidelines.md)
- What's already been built (completed task titles)

The implementor writes code only. No orchestration, no state management.

### Step 3: Check

The check command runs **on the host** (not in Docker), using the host's toolchain. A typical check command:

```bash
npm run lint && npm run typecheck && npm run build && npm test
```

The `&&` chain provides stop-on-failure for free — if lint fails, tests don't run. This replaces the old 7-stage oracle pipeline with a single deterministic command.

### Step 4: Pass / Retry / Blocked

- **Pass** → `update-tasks.py` marks the task `done`, git commits progress.
- **Fail** → One retry attempt. The retry prompt includes the check output so the LLM knows exactly what broke. The retry prompt is focused: "fix ONLY the errors shown, don't refactor or add features."
- **Still failing after retry** → Mark `blocked`, git commit, move to next task.

## Judge Review

After all tasks complete (or all remaining are blocked), the judge agent runs once. It evaluates **intent alignment**: did you build what the spec asked for, not just what the tasks described?

The judge receives:
- The full spec
- All tasks with their status
- A git diff stat of changes made

The judge is default-on. Opt out with `--no-judge`.

## In-Session Mode

`/trellis:implement <feature>`

The SKILL.md orchestrates within the user's conversation:
1. Pre-flight: read tasks.json, validate prerequisites
2. For each pending task: spawn test-writer agent → implement → run check → update tasks.json
3. After all tasks: spawn judge agent

This mode is interactive — the user can see progress and intervene. Good for features where you want to stay in the loop.

## Ralph Mode (Docker)

`/trellis:implement <feature> with ralph [--stream|--tail|--no-judge]`

The loop script (`scripts/ralph-loop.sh`) does ALL orchestration. The implement SKILL.md is NOT invoked inside Docker. Per task:

1. **`should-write-tests.py`** — Deterministic heuristic: should the test-writer run?
2. **`assemble-prompt.py test-writer`** → assembled prompt sent to `claude -p` in Docker
3. **`assemble-prompt.py implementor`** → assembled prompt sent to `claude -p` in Docker
4. **Check command** runs on host
5. **`update-tasks.py`** — marks task done or blocked
6. **Git commit** — each task is committed individually

Each Docker invocation gets a fresh context window. The LLM receives a fully-baked prompt with all context pre-assembled. No skill loading, no orchestration instructions, no state management.

### Output modes

| Flag | Behavior |
|------|----------|
| *(default)* | Silent — output goes to log files only. Status shown between tasks. |
| `--stream` | Full Claude output visible in real-time via `tee`, also logged. |
| `--tail` | Silent during task, shows last 50 lines of log after each task completes. |

### Authentication

Two modes, auto-detected:
- **API key** — Set `ANTHROPIC_API_KEY` env var.
- **OAuth/subscription** — Run `ralph-loop.sh --login` once. Credentials stored in a Docker volume and reused across tasks.

### Resume from interruption

Kill the process, re-run `/trellis:implement <feature> with ralph`. The script reads tasks.json and picks up from the first pending task. There is no separate state file — tasks.json on disk IS the resume point.

## Prompt Templates

Four plain-text files in `scripts/templates/` with `{{variable}}` placeholders:

- **test-writer.txt** — Writes failing tests for a task's verify criteria
- **implementor.txt** — Writes code for one task
- **implementor-retry.txt** — Fixes code after check failed
- **judge.txt** — End-of-run intent alignment review

`assemble-prompt.py` resolves the variables from spec artifacts (tasks.json, plan.md, spec.md, guidelines.md) and outputs the fully assembled prompt as JSON.

## When to Use Which Mode

| Scenario | Mode |
|----------|------|
| Small feature (1-3 tasks) | In-session |
| Want to stay in the loop | In-session |
| Large feature (5+ tasks) | Ralph |
| Fire-and-forget overnight | Ralph |
| Context degradation concern | Ralph |
| Docker not available | In-session |
